UI 组件库任选 Element Plus / Ant Design Vue（任选其一）；
Axios（统一封装请求/响应拦截器，注入 JWT）；

二、鉴权与角色（仅两类：普通/超级）
登录页使用 /auth/login 获取 JWT 与用户信息；
调用 /auth/profile 或 /rbac/me 拉取：userId, roleCode, isSuper, permissions[]；
路由守卫：
未登录跳转登录；
基于 isSuper 与 permissions[] 动态生成侧边菜单与按钮显隐；
普通管理员仅可使用被分配的功能点；超级管理员具有全部功能（但仍需要操作确认与审计）。
UI 上不要出现“超级管理员专属隐形能力”的提示文案，避免暗示越权。
三、全局交互与安全
确认流：对“高风险操作”采用分级策略：
L0 只读：无需确认；
L1 轻写：一次确认弹窗（展示参数摘要）
L2 重要改动：二次确认（再次明确影响范围）
L3 不可逆/对外影响：人工在环（弹出验证码或“二次输入 YES”+ 冷静倒计时）
通过后端返回的 ConfirmSheet（/ai/apply 在 mode=PLAN 时）渲染确认 UI；确认成功后再发起执行 API。
会话安全：收到后端判定“异常/恶意输入”的响应码或标识时：
立即终止当前会话（关闭流、禁用输入、toast 提示“已关闭会话（安全原因）”）；
清空当前上下文并记录在本地日志区
操作日志：管理端一切执行类操作的提交都带上 x-trace-id（前端生成或后端返回），并在本地控制台打印“操作摘要”，后端会统一写审计。
四、路由与页面（两种角色共用一套路由，按权限显隐）
基础布局：/login、/dashboard、/403、/404
AI 管理（/ai）：
对话页：输入框、模式切换（DIRECT/PLAN）、响应区（普通/流式）、引用来源折叠面板；
申请页：大输入框 → mode=PLAN → 展示 ConfirmSheet → “确认提交”后执行 → 展示“受理单号/结果”；
历史记录/会话列表（可选）。
管理员管理（超级管理员可见；普通管理员只读或不可见）：
管理员列表（增/改/停用、设置 roleCode、标记 isSuper 不可编辑或仅限超管）
普通用户列表（增/改/停用）
角色/权限（超级管理员可见）：
角色列表、角色绑定的权限点（资源-动作），批量授予/撤销；
将“现有功能点”映射为可配置项（菜单树 + 按钮级权限）。
现有业务模块（按权限显隐按钮）：
新闻、通知、景区、场馆、活动（含审核）、文件上传、系统监控（访问统计/系统指标/Redis 测试）。
日志与审计（超级管理员可见，普通管理员只读）：
操作日志列表：支持多条件筛选（操作者、模块、成功/失败、时间区间）与详情。
五、AI 相关交互（严格服务端执行）
对话（/ai/chat）
表单项：问题文本、多选“启用检索(RAG)”（默认关）、模式切换（DIRECT/PLAN）；
普通响应：Axios 请求展示为答复卡片；
流式响应（/ai/chat/stream）：使用 EventSource/SSE 渲染打字机效果；网络中断可自动重连 1~2 次；
展示“引用来源”（RAG 命中）在可折叠区。
申请（/ai/apply）
表单项：自由文本 + “模式=PLAN”强制；
第一步：发送 /ai/apply（mode=PLAN），后端返回 ConfirmSheet{action, summary, params, riskLevel}；
第二步：渲染确认弹窗（按 riskLevel 套用 L1/L2/L3 交互），用户确认后再发起“执行接口”（同一 /ai/apply 或后端返回的具体执行 API）；
执行结果以“受理单号/状态”卡片展示。
安全关闭
若返回“恶意/异常”标记（或特殊错误码），弹窗提示并禁用输入，引导用户返回首页或新建会话。
六、API 约定（可按后端实际命名替换）
认证：
POST /auth/login → {token, expireAt}
GET /auth/profile → {userId, username, roleCode, isSuper, permissions[]}
AI：
POST /ai/chat → {answer, citations?[]}
GET /ai/chat/stream?msg=...&mode=... → SSE 文本流
POST /ai/apply（mode=PLAN）→ {confirmSheet}；确认后再次调用或调用后端返回的执行 API → {ticketNo, status}
管理：
管理员：GET/POST/PUT /admin/admins、PUT /admin/admins/{id}/disable
用户：GET/POST/PUT /admin/users、PUT /admin/users/{id}/disable
角色：GET/POST/PUT /admin/roles、GET/PUT /admin/roles/{code}/permissions
日志：GET /admin/oplogs/page（分页）
业务（示例）：
新闻：/admin/news + /admin/news/page
通知：/admin/notices + /admin/notices/page
景区/场馆/活动/文件/监控等依此类推
请求头：Authorization: Bearer <token>；必要时加 x-trace-id。
错误约定：统一的 {code,message}；安全类错误增加 {reason: 'malicious'|'abuse'|'policy'} 便于前端做“关闭会话”。
七、通用组件（复用）
ConfirmDialog：渲染 ConfirmSheet（支持 L1/L2/L3，不同交互与样式）；
RiskTag：根据 riskLevel 显示标签；
OpBar：常用操作按钮条（增/改/删/导出/审核），自动根据权限按钮级显隐；
SseViewer：SSE 文本流展示组件；
SmartTable：表格 + 分页 + 条件查询（通用封装，传入列与数据源函数即可）；
UploadBox：文件上传与预览（后端返回 URL）；
Guard：指令或组件形式的“按钮级权限控制”（如 v-can="'NEWS:CREATE'"）。
八、前端安全与体验
Axios 拦截器：
请求：自动注入 token、trace-id；
响应：统一处理 401/403/安全关闭信号；
XSS/HTML：所有富文本使用安全渲染策略（只在需要时白名单）；
防抖/节流：对流式输入与频繁请求做节流；
失败兜底：网络中断、SSE 断流有 UI 反馈；
i18n/无障碍（可选）：至少保证键盘可达性与语义化标签。
